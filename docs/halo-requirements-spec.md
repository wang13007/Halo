# Halo 需求说明书

## 1. 文档目的

本文档基于当前代码实现整理 Halo 项目的现状需求说明，用于统一产品、开发、测试、实施和运维对系统能力边界的理解。本文档描述的是“已实现逻辑”，不包含尚未落地的规划能力。

## 2. 项目概述

Halo 是一个面向园区/楼宇能源管理场景的 Web 控制台，采用前后端分离部署模式：

- 前端：React + Vite
- 后端：Node.js + Express
- 数据层：Supabase PostgreSQL
- AI 能力：Google Gemini，可在未配置模型时回退为本地规则生成

系统当前重点围绕以下业务能力展开：

- 能耗概览与看板展示
- 能耗查询与智能问答
- 报告与诊断文本生成
- 系统接入信息浏览
- 会话历史持久化
- AI 辅助创建应用/组件元数据

## 3. 建设目标

### 3.1 业务目标

- 为项目/园区提供统一的能源数据查看入口。
- 让用户通过筛选条件和自然语言快速完成能耗查询。
- 借助 AI 将查询结果转化为分析结论、诊断内容或报告草稿。
- 提供系统集成、报告记录、项目数据的统一查看界面。

### 3.2 技术目标

- 通过统一 REST API 对外提供项目、能耗、报告、集成、会话能力。
- 通过 Supabase 落库存储核心业务数据与对话历史。
- 支持本地开发、Docker 部署、Render 部署。
- 支持通过 Excel 导入能耗查询数据。

## 4. 用户与使用方式

### 4.1 当前用户形态

根据现有代码，系统当前为单控制台使用模式，未实现真实登录、用户、角色、权限模型。

### 4.2 当前主要使用者

- 能源运营人员：查看看板、趋势、能耗明细、诊断结果。
- 实施/运维人员：初始化数据库、导入数据、检查集成状态。
- 管理人员：查看项目、报告、订阅展示页。

## 5. 功能范围

## 5.1 首页框架与导航

系统包含以下一级页面：

- `dashboard` 看板
- `chat` 智能对话
- `apps` 应用中心
- `config` 系统配置
- `subscription` 订阅方案

支持以下通用交互：

- 左侧导航切换页面
- 侧边栏展开/收起
- 深浅色模式切换
- 用户菜单展示

说明：

- “退出登录”仅为界面按钮，当前未接入认证注销逻辑。

## 5.2 看板模块

### 5.2.1 功能目标

用于展示能源运营关键指标卡片，并支持面板编辑。

### 5.2.2 当前已实现能力

- 内置两套看板标签：
  - 运营总览
  - 双碳监控
- 内置系统组件库，支持多种卡片尺寸：
  - `1:1`
  - `2:1`
  - `2:2`
- 支持以下编辑能力：
  - 修改当前看板名称
  - 新增/删除看板标签
  - 从组件库加入/移除组件
  - 拖拽排序
  - 调整组件尺寸
- 支持 AI Coding 生成新组件元数据并自动加入当前看板

### 5.2.3 数据与约束

- 当前看板组件内容主要为前端内置静态/示例数据。
- 看板编辑结果仅保存在前端内存状态中，刷新页面后不会持久化。
- AI 生成的是组件说明元数据，不会生成真实业务代码或落库记录。

## 5.3 智能对话模块

### 5.3.1 功能目标

通过“结构化筛选 + 自然语言提问”的方式完成能耗查询、对比、报告、诊断。

### 5.3.2 当前已实现能力

- 支持四类快捷意图：
  - 能耗查询
  - 能耗对比
  - 能源报表
  - 能源诊断
- 支持以下查询条件：
  - 项目
  - 能源类型
  - 统计粒度
  - 开始日期
  - 结束日期
  - 关键字
- 支持发送消息、停止当前生成、创建新会话、查看历史会话。
- 当前会话会自动保存到数据库。
- 点击历史会话后，可恢复消息内容和查询上下文。

### 5.3.3 后端处理逻辑

- 前端先调用 `/api/energy/query-config` 获取查询配置。
- 当意图为能耗相关时，前端将筛选条件转换为查询参数，调用 `/api/energy/query-report`。
- 查询结果作为 `dataPreview` 连同用户问题一起发送到 `/api/ai/chat`。
- 后端根据 `action/context/requestPayload/dataPreview` 生成回复。
- 对“能源报表/能源诊断”场景，后端会按规则自动选择报告模板。

### 5.3.4 会话持久化要求

系统需保存以下会话信息：

- 消息列表
- 标题
- 摘要
- 状态
- 最近消息时间
- 查询上下文

当前上下文中实际保存的字段包括：

- 项目名
- 组织 ID
- 能源类型显示值与原始值
- 粒度显示值与原始值
- 起止日期
- 时间范围文案
- 查询关键字
- 来源标识 `chat-workspace`

## 5.4 能耗分析模块

### 5.4.1 功能目标

展示项目能耗统计、趋势和分项占比。

### 5.4.2 当前已实现能力

- 从 `/api/energy/analysis` 读取数据。
- 展示以下统计卡片：
  - 今日总能耗
  - 本月累计
  - 预估费用
  - 碳排放量
- 展示以下图表：
  - 24 小时用电趋势堆叠柱图
  - 分项能耗占比饼图

### 5.4.3 统计口径

后端基于 `energy_metrics` 表计算：

- 今日与昨日对比
- 本月与上月对比
- 当月费用累计
- 当月碳排累计
- 当日按 `source` 的时序图
- 当月按 `source` 的分项汇总

说明：

- 当前数据来源为数据库，不是直接对接实时 EMS 流式数据。
- 前端内置了兜底展示数据，当接口不可用时可继续展示示例图表。

## 5.5 应用中心模块

### 5.5.1 功能目标

提供收藏应用、我的应用、应用市场三类视图，并支持 AI 辅助创建应用。

### 5.5.2 当前已实现能力

- 标签页：
  - 收藏应用
  - 我的应用
  - 应用市场
- 操作能力：
  - 收藏/取消收藏
  - 安装
  - 卸载
  - 运行
  - 编辑标题与描述
  - 复制应用
  - AI 创建应用元数据

### 5.5.3 当前边界

- 应用中心数据为前端本地状态，未接入数据库。
- “运行”当前仅反馈状态提示，不会打开真实业务应用。
- AI 创建应用只会返回标题、描述、摘要、要点等元数据，不生成独立页面或服务。

## 5.6 系统配置模块

### 5.6.1 功能目标

集中查看 EMS、IBMS、Supabase 的系统信息、接口清单和近期记录。

### 5.6.2 当前已实现能力

- 预置三张系统卡片：
  - EMS 系统
  - IBMS 系统
  - Supabase 数据层
- 点击卡片后弹出详情窗口。
- 详情窗口会并行读取：
  - `/api/health`
  - `/api/integrations`
  - `/api/reports`
  - `/api/projects`
- 按不同系统展示请求记录：
  - Supabase：展示报告记录
  - EMS/IBMS：展示系统集成记录

### 5.6.3 当前边界

- 当前模块为“查看型配置中心”，未提供在线新增/编辑集成配置的界面。
- 后端虽提供创建集成 API，但前端当前未提供录入表单。

## 5.7 订阅方案模块

### 5.7.1 当前已实现能力

- 展示基础版与高级版套餐
- 支持月/季/年切换展示价格

### 5.7.2 当前边界

- 仅为前端展示页
- 未接入订单、支付、账户授权、套餐生效逻辑

## 5.8 报告与诊断生成

### 5.8.1 功能目标

根据能耗查询结果自动生成 Markdown 报告或诊断内容。

### 5.8.2 当前已实现逻辑

- 支持按场景自动选择模板：
  - 集团日报/周报/月报/季报/年报
  - 项目日报/周报/月报/季报/年报
  - 单电表用能异常诊断模板
- 支持判断数据可用性，自动跳过缺少支撑数据的章节。
- 未配置 Gemini 时，系统使用本地规则生成结构化 Markdown 文本。

### 5.8.3 生成规则要求

- 只能使用当前请求中的已知数据生成内容。
- 不允许虚构天气、客流、面积、设备运行状态、预算目标等缺失数据。
- 必须明确说明哪些分析项因数据不足被跳过。

## 6. 接口需求

当前系统已实现的主要接口如下：

| 接口 | 方法 | 用途 |
| --- | --- | --- |
| `/api/health` | GET | 健康检查，验证数据库配置和可达性 |
| `/api/projects` | GET | 查询项目列表 |
| `/api/chat/sessions` | GET | 查询会话历史 |
| `/api/chat/sessions/:sessionId` | GET | 查询单个会话详情 |
| `/api/chat/sessions` | POST | 新建会话 |
| `/api/chat/sessions/:sessionId` | PATCH | 更新会话 |
| `/api/energy/analysis` | GET | 查询能耗分析数据 |
| `/api/energy/query-config` | GET | 查询能耗筛选配置 |
| `/api/energy/quick-projects` | GET | 查询项目快捷列表 |
| `/api/energy/query-report` | POST | 查询已导入的能耗明细 |
| `/api/energy/query-report-legacy` | POST | 代理旧版 Longfor 查询接口 |
| `/api/energy/metrics` | POST | 写入能耗指标 |
| `/api/ai/chat` | POST | AI 对话生成 |
| `/api/ai/coding` | POST | AI 生成应用/组件元数据 |
| `/api/reports` | GET | 查询报告记录 |
| `/api/reports` | POST | 新增报告记录 |
| `/api/integrations` | GET | 查询系统集成记录 |
| `/api/integrations` | POST | 新增系统集成记录 |

## 7. 数据需求

系统当前依赖以下核心数据对象：

- 项目
- 系统集成
- 能耗指标
- 能耗查询明细
- 分析报告
- 聊天会话

其中：

- `energy_metrics` 用于首页统计与趋势分析
- `energy_query_records` 用于结构化能耗查询
- `chat_sessions` 用于对话持久化
- `projects.metadata` 用于保存导入项目的查询配置扩展字段

## 8. 非功能需求

### 8.1 性能

- 查询接口默认分页返回，能耗查询默认每页 20 条。
- 能耗查询后端按 1000 条批次读取数据库并再过滤。
- `/api/health` 应可作为部署平台健康检查接口。

### 8.2 可维护性

- 通过 SQL migration 管理数据库结构。
- 通过独立脚本完成数据库初始化和 Excel 数据导入。
- 报告模板以 Markdown 文件方式维护，便于调整。

### 8.3 可靠性

- 未配置 AI Key 时，AI 能力可回退到本地规则输出。
- 前端部分页面具备示例数据兜底展示能力。

### 8.4 安全性

- 服务端通过 `SUPABASE_SERVICE_ROLE_KEY` 访问数据库。
- `SUPABASE_SERVICE_ROLE_KEY` 不应暴露到前端。
- CORS 白名单可通过环境变量控制。

## 9. 当前版本限制

- 未实现真实用户认证、RBAC、组织隔离。
- 看板、应用中心、订阅页主要为前端态，未持久化。
- 未实现文件上传式报表生成与下载闭环。
- 未实现系统配置页面的新增、编辑、删除集成功能。
- RLS 已开启，但迁移中未定义策略，当前依赖服务端 Service Role 访问。
- 能耗查询接口当前只读取 Supabase 数据，不在 API 层启用本地缓存回退。

## 10. 后续扩展建议

- 增加用户、角色、权限和登录体系。
- 将看板配置、应用中心配置持久化到数据库。
- 增加报表文件生成、归档和下载能力。
- 在配置中心补齐集成配置维护表单。
- 为查询、报告、诊断建立审计日志与操作留痕。
